# 质量标准

## 四维评估体系

Evaluator Agent 使用以下四个维度评估模块实现质量。每个维度独立评分，任一维度未达阈值则评估不通过，需要 Generator 根据反馈迭代。

## 1. 正确性（权重 40%，阈值：100% 测试通过）

**核心问题**：实现是否正确完成了规格中定义的所有行为？

评估要点：
- 所有单元测试通过
- 覆盖规格文档中定义的每个功能点
- 边界条件测试：空输入、极值、溢出、零值
- 异常路径测试：无效参数、非法状态转换
- 确定性验证：相同输入始终产生相同输出

不通过的信号：
- 任何测试失败
- 规格中定义的功能缺少对应测试
- 仅测试正常路径，缺少边界和异常用例

## 2. 架构合规（权重 25%，阈值：0 违规）

**核心问题**：代码是否遵循 architecture.md 中定义的分层和依赖规则？

评估要点：
- 运行 `python -m harness.lint.check_architecture` 零违规
- 模块目录结构符合规范（models.py, interfaces.py, v1_readable/engine.py）
- 引擎类实现了对应的接口
- 无跨模块直接导入
- 依赖方向正确（Models → Interfaces → Engine → Simulation）

不通过的信号：
- 架构检查报告任何违规
- 缺少必要文件（如 interfaces.py）
- 引擎类未继承接口
- 存在跨模块 import

## 3. 代码质量（权重 20%，阈值：≥7/10）

**核心问题**：代码是否清晰、可维护、符合项目风格？

评分细则（各 1 分，满分 10 分）：
1. 类和函数有清晰的单一职责
2. 命名准确表达意图（无 `data`、`info`、`temp` 等模糊命名）
3. 函数长度合理（单函数不超过 50 行）
4. 关键参数和返回值有类型标注
5. Pydantic 模型使用恰当（字段有类型约束和默认值）
6. 无硬编码魔法数字（使用常量或配置）
7. 错误处理得当（使用自定义异常，不吞掉异常）
8. 无重复代码（DRY 原则）
9. 公共方法有简洁的 docstring
10. 模块的 `__init__.py` 导出了公共 API

## 4. 性能（权重 15%，阈值：基准达标）

**核心问题**：实现是否在合理的性能范围内？

评估要点：
- 基准测试存在且可运行（`tests/modules/{name}/test_benchmark.py`）
- v1_readable 实现的性能在可接受范围内（具体阈值由规格定义）
- 无明显的 O(n²) 或更差的算法复杂度（除非规格明确要求）
- 无不必要的对象创建或深拷贝

v2_optimized（如存在）的额外要求：
- 性能优于 v1_readable 至少 2x
- 通过 pytest-benchmark 验证

## 评估报告格式

Evaluator 输出的评估报告应遵循以下结构：

```markdown
# 评估报告：{module_name}

## 总结
- 状态：通过 / 不通过
- 总分：{score}/100

## 各维度评分

### 正确性（{score}/40）
- 测试结果：{passed}/{total} 通过
- [具体发现...]

### 架构合规（{score}/25）
- 违规数：{count}
- [具体违规...]

### 代码质量（{score}/20）
- 评分：{score}/10 × 2
- [具体问题...]

### 性能（{score}/15）
- 基准测试：通过/不通过
- [具体数据...]

## 改进建议
1. [可操作的具体建议...]
```
