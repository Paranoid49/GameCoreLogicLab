# 质量标准

## 四维评估体系

Evaluator Agent 使用以下四个维度评估模块实现质量。每个维度的权重为基准值，Planner 可在规格文档的"评估侧重"章节中按模块特点调整。

## 权重分配

| 维度 | 基准权重 | 可调范围 | 说明 |
|------|---------|---------|------|
| 正确性 | 40% | 30-60% | 算法密集型模块可提高 |
| 架构合规 | 25% | 15-30% | 所有模块不可低于 15% |
| 代码质量 | 20% | 10-25% | — |
| 性能 | 15% | 10-30% | 性能敏感型模块可提高 |

如果规格中未定义"评估侧重"，使用基准权重。

## 评估方式

Evaluator 的核心工作不是"打分"，而是**独立验证 + 主动攻击**：

1. **运行 Generator 的测试**：作为基线，但不足以判定通过
2. **独立构造验证**：Evaluator 自己编写探测性测试，覆盖 Generator 可能遗漏的边界
3. **不变量检查**：验证核心算法的数学/逻辑不变量
4. **随机模糊测试**：用随机输入压力测试引擎
5. **规格逐条对照**：手动验证每条验收标准是否真的被满足

## 判定标准

- **PASS**：零致命 Bug + 架构零违规 + 独立验证无致命发现
- **FAIL**：输出按严重度排序的 Bug 列表，每条含复现方式和修复建议

## 1. 正确性（权重 40%，阈值：100% 测试通过）

**核心问题**：实现是否正确完成了规格中定义的所有行为？

评估要点：
- 所有单元测试通过
- 覆盖规格文档中定义的每个功能点
- 边界条件测试：空输入、极值、溢出、零值
- 异常路径测试：无效参数、非法状态转换
- 确定性验证：相同输入始终产生相同输出

不通过的信号：
- 任何测试失败
- 规格中定义的功能缺少对应测试
- 仅测试正常路径，缺少边界和异常用例

## 2. 架构合规（权重 25%，阈值：0 违规）

**核心问题**：代码是否遵循 architecture.md 中定义的分层和依赖规则？

评估要点：
- 运行 `python -m harness.lint.check_architecture` 零违规
- 模块目录结构符合规范（models.py, interfaces.py, engine.py）
- 引擎类实现了对应的接口
- 无跨模块直接导入
- 依赖方向正确（Models → Interfaces → Engine → Simulation）

不通过的信号：
- 架构检查报告任何违规
- 缺少必要文件（如 interfaces.py）
- 引擎类未继承接口
- 存在跨模块 import

## 3. 代码质量（权重 20%，阈值：≥7/10）

**核心问题**：代码是否清晰、可维护、符合项目风格？

评分细则（各 1 分，满分 10 分）：
1. 类和函数有清晰的单一职责
2. 命名准确表达意图（无 `data`、`info`、`temp` 等模糊命名）
3. 函数长度合理（单函数不超过 50 行）
4. 关键参数和返回值有类型标注
5. Pydantic 模型使用恰当（字段有类型约束和默认值）
6. 无硬编码魔法数字（使用常量或配置）
7. 错误处理得当（使用自定义异常，不吞掉异常）
8. 无重复代码（DRY 原则）
9. 公共方法有简洁的 docstring
10. 模块的 `__init__.py` 导出了公共 API

## 4. 性能（权重 15%，阈值：基准达标）

**核心问题**：实现是否在合理的性能范围内？

评估要点：
- 基准测试存在且可运行（`tests/modules/{name}/test_benchmark.py`）
- 实现的性能在可接受范围内（具体阈值由规格定义）
- 无明显的 O(n²) 或更差的算法复杂度（除非规格明确要求）
- 无不必要的对象创建或深拷贝
- 通过 pytest-benchmark 验证

## 评估报告格式

Evaluator 输出 Bug 列表式报告，而非数值评分。详见 `.claude/agents/evaluator.md` 中的报告模板。

核心结构：
- **判定**：PASS / FAIL
- **Bug 列表**：按致命/重要/建议分级，每条含文件路径、复现方式、修复建议
- **独立验证结果**：边界攻击、不变量检查、随机模糊测试的发现
- **规格对照**：验收标准逐条验证状态
